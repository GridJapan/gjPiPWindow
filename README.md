# gjPiP

任意のディスプレイ（`FreeDisplay Virtual` などの仮想ディスプレイを含む）を、小さな
ウィンドウにリアルタイム表示する macOS メニューバーアプリ。BetterDisplay Pro の
Picture-in-Picture 相当を、FreeDisplay と併用する前提で自作したもの。
「常に最前面」はメニューで切り替えられる（既定は OFF）。

PiP ウィンドウ上でのマウス操作は、元のディスプレイにそのまま転送される。

## ビルド

```sh
./make-signing-cert.sh   # 最初の一度だけ
./build.sh               # → build/gjPiP.app
open build/gjPiP.app
```

`swift build` 単体でもバイナリはできるが、TCC（画面収録・アクセシビリティ）は
**署名済みの .app バンドルにしか権限を与えない**ため、必ず `build.sh` を使う。

デバッグログは `GJPIP_DEBUG=1` で出る。外から挙動を確かめにくいアプリなので
（スクリーンショットを撮る側にも画面収録権限が要るし、操作モードはマウスを奪う）、
実質これが唯一の観測手段になる。

```sh
open --env GJPIP_DEBUG=1 --stderr /tmp/gjpip.log build/gjPiP.app
```

## 必要な権限

| 権限 | 用途 | 設定場所 |
|---|---|---|
| 画面収録 | ディスプレイのキャプチャ | プライバシーとセキュリティ → 画面収録とシステムオーディオ録音 |
| アクセシビリティ | マウス操作の転送（イベントタップ） | プライバシーとセキュリティ → アクセシビリティ |

画面収録は初回起動時にダイアログが出る。アクセシビリティは、PiP を初めてクリックして
操作モードに入ろうとしたときに要求される。**どちらも許可後にアプリの再起動が必要。**

許可ダイアログは**マウスのある画面ではなく、どこか別のディスプレイに出ることがある**。
多画面環境では見落としやすい。

`CGRequestScreenCaptureAccess()` は**プロンプトを非同期で出して即座に false を返す**。
false = 拒否された、ではない。「まだ答えていない」だけなので、ここを誤読すると
「プロンプトが出ていない」と誤診する。

### 一覧に載っているのに許可が効かないとき

System Settings の一覧でトグルが ON なのにアプリ側が未許可、という状態が起きる。
署名 identity が変わると、TCC に残った古い csreq と現在のアプリが一致しなくなるため。
**トグルを一度 OFF にしてから ON に戻す**と、現在の要件で記録し直される
（`tccutil reset ScreenCapture co.gridworld.gjPiP` だけでは直らないことがある）。

### なぜ自己署名証明書が要るか

アドホック署名（`codesign -s -`）だと designated requirement が cdhash になる：

```
designated => cdhash H"e5f648f4ab74180e5809ed375cd52605eb08a71b"
```

TCC はこれで identity を判定するので、**コードを1行変えてリビルドしただけで別アプリ扱いになり、
権限が全部外れる**（実測で確認済み）。毎回 System Settings で再許可 + Touch ID になり、
開発が成立しない。

`make-signing-cert.sh` が作る自己署名証明書で署名すると、要件が証明書ベースになる：

```
designated => identifier "co.gridworld.gjPiP" and certificate root = H"9414593b..."
```

cdhash が含まれないので、リビルドしても権限が維持される。証明書の秘密鍵はログイン
キーチェーンにだけ入り、リポジトリには何も置かない。

証明書を作り直した場合は要件が変わるので、System Settings で許可し直しになる。

## 使い方

メニューバーの PiP アイコンから：

- **ディスプレイ** — 選ぶと PiP ウィンドウが開く。もう一度選ぶと閉じる。複数同時に開ける
- **PiP ウィンドウを開く画面** — 新しい PiP がどのディスプレイに出るか。既定は BenQ EX3410R。
  選択は記憶される。ディスプレイ ID は再接続で変わるので**名前で**保存している
- **枠の外に出たら解除する辺** — 上下左右それぞれで、操作モードの出口にするかを切り替える。
  既定は**下だけ OFF**（後述）
- **常に最前面** — 既定は OFF。ON にすると他のウィンドウに隠れなくなるが、
  **Mission Control（Control + ↑）に出なくなる**（後述）
- **フレームレート** — 30 / 60 / 120 fps。負荷が気になるなら下げる
- ウィンドウはタイトルバーでドラッグ、端でリサイズ（アスペクト比は固定）

### 操作モード

**フォーカスされている PiP の中をクリックすると操作モードに入る。**

非フォーカス状態からの1クリック目は、他のウィンドウと同じくフォーカス取得に使われる。
マウスを別のディスプレイへ持って行かれるのは、ただ手前に出したかっただけの人には
大きすぎる出来事なので、2クリック目という明確な意思表示を要る形にしてある。

カーソルが対象ディスプレイへ移り、以降は物理マウスがそのままそちらを操作する。
抜け方は3つ:

| 抜け方 | 用途 |
|---|---|
| **端まで動かす** | 通常。モニタから出るのと同じ感覚で、カーソルは PiP の枠の外に出る |
| **Esc 5回連打** | 緊急脱出。コードを覚える必要が無い |
| **Control + Command + Esc** | 予備 |

### なぜ下端だけ既定で OFF なのか

**下端は Dock を出す場所だから。** そこを「触れたら追い出される辺」にすると、
Dock に手を伸ばす動作そのものが脱出動作になり、Dock には永久に届かない。
macOS には Dock の反応距離を変える設定が無い（`autohide-delay` は待ち時間であって距離ではない）
ので、こちら側で辺を空けるしかない。

辺は**扉か壁か**として扱う。

| | 隣にディスプレイがある | 隣に何も無い |
|---|---|---|
| **扉（ON）** | カーソルが自然に出ていく → 解除 | 押し当てると解除 |
| **壁（OFF）** | 出ようとしたら押し戻す | macOS が既に止めている |

「壁」で押し戻すのが要る理由は、そこを省くと**カーソルだけ隣の画面へ抜けて、
PiP が操作モードのまま取り残される**から。

ドラッグ中は端でも解除されない（掴んでいる物を落とさないため）。
どうしても復帰しないときは `pkill -f gjPiP.app` でプロセスごと落とせば、
イベントタップが外れてマウスは戻る。

## 設計メモ

### なぜイベントタップなのか

「PiP 内のクリック座標を変換して `CGEventPost` する」だけでは動かない。合成イベントを
1つ送った時点で本物のカーソルが対象ディスプレイへワープし、以降 PiP ウィンドウは
`NSEvent` を受け取らなくなるため、2回目以降の座標を取る手段が無くなる。

答えは「転送しない」こと。**カーソルを対象ディスプレイに置いたら、あとは放っておく**。
物理マウスがそのままネイティブに、通常の速度と加速度でそれを動かす。
`InteractionController` がやるのは境界の監視だけで、`cgSessionEventTap` で各イベントの
移動先を見て、対象ディスプレイの外に出た瞬間に解除して PiP の枠の外にカーソルを置く。

### delta を積算してはいけない（実際にやらかした）

最初の実装は `mouseEventDelta` を自前の仮想座標に積算していた。これは壊れる。
**自分で起こしたカーソルのワープが mouseMoved として跳ね返り、その delta を
自分で積算してしまう**ため。ログに残った実例:

```
enter interaction, normalized (0.507, 0.5) → (6333.5, 600.0) on display 11
leaving  display 11 at (7647.5, -975.0)          ← 0.137秒後、誰も動かしていないのに
```

差分 (+1314, −1575) は、クリック地点 (5016,2176) からワープ先 (6333.5,600) までの
距離そのもの。カーソルがどこへ行くかは OS が計算済みなので、`event.location` を
読むのが唯一正しい。積算をやめた結果、仮想座標・magic マーカー・クリックの複製が
まとめて不要になり、コードは短くなった。

### NSPanel は Esc で閉じる

`NSPanel` は Esc を「キャンセル」と解釈して自分を閉じる（`cancelOperation:`）。
PiP をクリックするとパネルがキーウィンドウになるため、**Esc 5回連打の1発目で
窓が消える**という形で緊急脱出が丸ごと無効化されていた。しかも当時は窓が閉じても
イベントタップは生きたままで、マウスを掴まれたまま PiP だけ消えるという最悪の状態だった。

対策は `cancelOperation:` の空実装。当初は `canBecomeKey` を `false` にする方で塞いだが、
後述の理由でそれは撤回した。緊急脱出そのものはグローバルなイベントタップで Esc を読んでいて、
窓のキー状態とは無関係なので、この一箇所で足りる。

加えて、掴んだまま PiP が閉じられたら必ず解除するようにした
（`releaseMouseIfCaptured`）。見えないディスプレイにカーソルを置き去りにして、
歩いて出るための枠まで消える状況を作ってはいけない。

### 「フォーカスを取らない窓」は前面に出られない

`canBecomeKey` を `false`、`canBecomeMain` も既定の `false`（`NSPanel` の既定）、
さらに `.nonactivatingPanel` ── この三点セットで**フォーカスを一切取らない窓**にしていた。
常に最前面に浮いている間は、これで何の問題も無かった。

`level = .normal` にした途端に破綻した。**キーにも main にもなれない窓しか持たないアプリは、
アクティブになれない**。実測:

```
activate() returned: true
isActive: false          ← 活性化を要求しても、アクティブにならない
```

そして macOS が通常レベルのウィンドウを前面に出す手段は「そのアプリをアクティブにすること」
しかない。つまり **PiP は「永久に前面に出られない窓」**になっていた。Mission Control で選ぶと
一瞬持ち上がってから、macOS が直前のアプリをアクティブに戻し、その窓に埋め戻される。

なので今は普通に `canBecomeKey` / `canBecomeMain` を `true` にし、`.nonactivatingPanel` も
外している。失うものは「PiP がフォーカスされている間のキー入力が gjPiP に行く」ことだけで、
映している画面を操作したければそこをクリックすれば、その画面のアプリにフォーカスが移る。

### カーソルを隠してはいけない

`NSCursor.hide()` を入れていたが、これが「マウスが死んだ」の正体だった。
仮想ディスプレイには物理モニタが無いので、**そこへ飛んだカーソルを見る手段は
PiP の映像だけ**。それを隠せばどこにも見えなくなる。

### 最初のクリックは、あえて取りこぼす

`PiPContentView` は `acceptsFirstMouse` に `false` を返している。**非アクティブな
ウィンドウへの最初のクリックは、フォーカス取得に消費されて view に届かない** ── その
既定の挙動に、あえて乗っている。

以前は逆に `true` を返していた。PiP はキーウィンドウになれない窓だったので、そうしないと
クリックがどこにも届かず「クリックしても何も起きない」窓になったからだ。フォーカスを取れる
ようにした結果、その必要が消えただけでなく、**害のほうが目立つようになった**。マウスを
別のディスプレイへ持って行かれるのは、ただ PiP を手前に出したかっただけの人には大きすぎる
出来事で、それは2クリック目という明確な意思表示に値する。

### 常に最前面と Mission Control は両立しない

Mission Control が並べるのは `.managed` な通常レベルのウィンドウだけ。「常に最前面」は
その真逆 ── `level = .floating` は通常レベルより上で対象外、`.canJoinAllSpaces` は
「全 Space に居る」窓（メニューバーと同じ扱い）で、どの Space に属するかという概念自体が
無いため、Space を俯瞰する画面に置き場所が無い。

なので `setAlwaysOnTop(_:)` は **level と collectionBehavior をまとめて入れ替える**。
level だけ戻して `.canJoinAllSpaces` を残すと、「最前面ではないのに Mission Control にも
出ない」という、どちらの利点も無い状態になる。

### NSPanel は contentRect の origin を尊重しない

`NSPanel(contentRect:styleMask:backing:defer:)` は、渡した origin を**イニシャライザの
時点で勝手に書き換える**。実測:

```
指定 contentRect        = (2696.0, 24.0, 720.0, 450.0)
panel after super.init  = (456.0, 24.0, 720.0, 474.0)   ← x だけ動かされた
```

y と幅はそのまま、x だけ。`contentAspectRatio` でもその後の設定でもなく init 自身。
なので init の最後で `setFrameOrigin(frameRect(forContentRect:).origin)` を明示的に
呼んで置き直している。これが無いと「開く画面の指定」が効かない。

### 描画

`SCStream` が返す `CMSampleBuffer` から `IOSurface` を取り出し、`CALayer.contents` に
直接渡すゼロコピー。Metal もシェーダも要らない。ウィンドウのアスペクト比を
ソースに固定しているのでレターボックスが出ず、view 座標 → ディスプレイ座標の
変換が単純な正規化だけで済む。

`SCFrameStatus.complete` 以外のフレームは捨てている（画面が変化していないとき、
ストリームは前フレームを非 complete で繰り返す）。

## 動作確認のしかた

このアプリは外側から観察しづらく、素朴な確認方法がことごとく嘘をつくので注意。

**`screencapture` は使えない。** PiP に何が映っているかを撮ろうとしても、撮る側の
プロセス（ターミナル等）にも画面収録権限が要る。無いと
`could not create image from display` で失敗する。代わりに `GJPIP_DEBUG=1` の
フレームログで確認する。

**System Events の `click at` では検証できない。** アクセシビリティ API 経由の擬似
クリックなので、AX アクションを持たないカスタム view には**届かない**。`mouseDown` が
呼ばれず「アプリのバグ」に見えるが、嘘をついているのはテストのほう。本物の HID
クリックが要る：

```swift
CGEvent(mouseEventSource: nil, mouseType: .leftMouseDown,
        mouseCursorPosition: p, mouseButton: .left)?.post(tap: .cghidEventTap)
```

**ターミナルから直接バイナリを起動すると TCC の判定が変わる。** 責任プロセスが
ターミナルとみなされるため、`./build/gjPiP.app/Contents/MacOS/gjPiP` と直に叩くと
権限が無い扱いになる。必ず `open` 経由で起動して確認する。

**コメントを足しただけではリビルドの検証にならない。** バイナリが変わらないので
cdhash も変わらず、「権限が維持された」ように見えるだけ。署名 identity の検証には
実際にコードを変える必要がある。

**操作モードで詰まったら `pkill -f gjPiP.app`。** プロセスが死ねばイベントタップも
外れ、マウスは戻る。
